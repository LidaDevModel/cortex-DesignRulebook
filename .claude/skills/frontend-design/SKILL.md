# Frontend Design

Generate creative, production-grade frontend code with exceptional aesthetic quality. Every interface should feel intentionally designed — not templated.

## Step 0: Contract Check

If `VISION.md` exists at the project root, read it before anything else. It is the binding project contract: its tokens, type ramp, signature technique, layout pattern, exact wording tables, and animation timings **override** both the taste profile and the guidelines below. Run its pre-edit checklist for changes to existing surfaces. The creative guidance in this skill applies fully on greenfield projects and wherever the contract is silent.

## Step 0.5: Reuse Check

Before building any component, honor the Reuse Gate (`design-intent` rule): find how the project already solves this. If `COMPONENTS.md` exists at the root (generated by `npx design-fleet inventory`), search it **by the problem** — a data table? a date range? a modal? — and reuse or extend the existing component instead of writing a new one. Re-deriving a near-duplicate fragments the system. Build new only when nothing covers it; if an existing component is deficient, improve it in place rather than forking. Regenerate `COMPONENTS.md` after adding a reusable component so the next session inherits it.

## When to Use

Use when building, creating, or implementing UI — components, pages, layouts, full applications. This is the primary skill for going from intent to production code.

This skill also owns motion: animation timing, easing, springs, exit animations, and advanced CSS/DOM patterns (pseudo-elements, View Transitions, container animation, morphing icons, predictive prefetching) live in the **Motion & advanced patterns** section below and its reference files.

**Not this skill:** If they want to review existing UI quality (including formal Nielsen heuristic reviews), use `design-review`. If they want a codebase audit or a project VISION.md, use `design-contract`.

## Design Thinking

Before coding, commit to a clear direction:

- **Purpose**: What problem does this solve? Who uses it?
- **Tone**: Pick a strong aesthetic — brutally minimal, maximalist, retro-futuristic, organic, luxury, playful, editorial, brutalist, art deco, soft/pastel, industrial, etc. Use these as inspiration but design something true to the chosen direction.
- **Differentiation**: What makes this unforgettable? What's the one thing someone remembers?

**Bold maximalism and refined minimalism both work — the key is intentionality, not intensity.** Match implementation complexity to the vision: maximalist designs need elaborate effects; minimal designs need precision in spacing, typography, and subtle details.

## Aesthetics Guidelines

### Typography

Choose what the user wants first.

→ *If using Next.js + Tailwind, consult [nextjs-tailwind reference](references/nextjs-tailwind.md) for `next/font` loading and CSS variable scoping.*

- Use `font-display: swap` in all font declarations so text renders immediately
- Self-host fonts or use the framework's font optimization (Next.js `next/font`, Nuxt `@nuxt/fonts`, etc.)
- Never default to Inter, Roboto, Arial, or system fonts. Never converge on the same font (e.g. Space Grotesk) across designs. Every project gets a unique pairing.

### Color & Theme

Commit to a cohesive palette. Dominant colors with sharp accents outperform timid, evenly-distributed palettes.

→ *If using Tailwind, consult [nextjs-tailwind reference](references/nextjs-tailwind.md) for CSS variable wiring and token patterns.*

- Define colors as CSS custom properties in your global stylesheet and wire them through your framework's design token system
- Use semantic names (`--color-surface`, `--color-primary`, `--color-muted`) not raw hex in components
- Vary between light and dark themes across designs. Avoid cliched schemes (especially purple gradients on white).

### Color & Theme (Deep)

→ *Consult [color-and-contrast reference](references/color-and-contrast.md) for OKLCH implementation, tinted neutral palettes, and dark mode code patterns. See `color-contrast.mdc` for the underlying rules.*

### Motion & Interaction

→ *Consult [interaction-design reference](references/interaction-design.md) for 8 states, focus rings, and dropdown positioning. For the full motion decision framework (easing, timing, springs, exits, performance, reduced motion) and the advanced-pattern catalog, see the [Motion & advanced patterns](#motion--advanced-patterns) section below.*

Focus on high-impact moments: one well-orchestrated page load with staggered reveals creates more delight than scattered micro-interactions.

- **CSS-first**: Use CSS transitions and `animation-delay` for staggered reveals, hover states, scroll-triggered effects
- **Motion library** (Framer Motion / Motion): Use for complex orchestration in React when CSS alone falls short
- Surprise with scroll-triggered animations and hover states that feel tactile
- **Design all 8 states**: default, hover, focus, active, disabled, loading, error, success — for every interactive element
- **Use native APIs**: `<dialog>` for modals, `popover` for tooltips/dropdowns, `inert` for background content

### Spatial Composition

Break predictable layouts. Use asymmetry, overlap, diagonal flow, grid-breaking elements, generous negative space OR controlled density — whatever serves the aesthetic.

### Backgrounds & Texture

Create atmosphere. Gradient meshes, noise textures, geometric patterns, layered transparencies, dramatic shadows, grain overlays, decorative borders — choose what fits the tone. Never default to flat solid-color backgrounds when the design calls for depth.

### Motion decisions (Layer 1)

Every animation decision starts with four questions: **entering or exiting?** → `ease-out`. **An on-screen element moving?** → `ease-in-out`. **A hover/color transition?** → `ease`. **Will users see it 100+ times daily?** → don't animate it. Springs (not easing) for gesture-driven, interruptible motion that must preserve velocity. UI animations stay under 300ms; exits run ~20% faster than entrances; if it feels slow, shorten the duration before touching the curve. The frequency rule overrides everything. Every animated element needs a `prefers-reduced-motion` opt-out that disables it entirely. A project VISION.md animation-timing table overrides the generic durations here.

## Project Integration Rules

These rules ensure the skill's creative output integrates cleanly with existing project architecture.

### Styling

Use the project's utility/token system consistently. Avoid inline styles for properties the styling framework can handle. Keep raw CSS minimal — reach for it only when the framework cannot express the effect (complex gradients, `@keyframes`, noise SVG filters, pseudo-element tricks).

→ *If using Tailwind + cn(), consult [nextjs-tailwind reference](references/nextjs-tailwind.md) for conditional class patterns and inline style avoidance.*

### Client / Server Boundary (SSR frameworks)

→ *If using Next.js RSC, consult [nextjs-tailwind reference](references/nextjs-tailwind.md) for `'use client'` boundary rules.*

For any SSR framework: push interactivity to the smallest component that needs it. Static markup, layout, and data fetching stay on the server. Only components with event handlers, hooks, or browser APIs need client-side hydration.

### Component Structure

- Place components in `components/[feature]/` following the project's feature-based organization
- Aim for <200 lines per component — extract sub-components for complex layouts
- One component per file (small helpers co-located are fine)
- Use PascalCase for component files, kebab-case for directories

### Shared UI Library

Check if the project has a shared UI library (`@your-org/ui` or similar) before creating new base components. Use existing shared components as foundations and customize via `className` overrides, composition, or variant props. Only create new components in `components/ui/` when the shared library doesn't cover the need.

### Scrollbar Stability

**When the main container can toggle between overflowing and not** → set `overflow-y: scroll` so the scrollbar always reserves its space and content never shifts. Make it invisible by default (transparent track and thumb in global CSS) and reveal only a thin, pastel-colored thumb on hover. Do not use `scrollbar-gutter: stable` — it reserves the browser's default scrollbar width, which mismatches custom thin overrides (`::-webkit-scrollbar { width: 6px }`).

### Cursor Affordance

All interactive elements (`button`, `a`, `[role="button"]`, `select`, `summary`) must show `cursor: pointer` on hover. Set this as a global rule in `globals.css` rather than per-component — one declaration covers the entire app. Never leave a clickable element with the default arrow cursor.

### Text Truncation in Tables

**When a table cell's text can exceed its column width** → `<td>` ignores `overflow: hidden`, so truncate on an inner wrapper: a `<div className="truncate">` inside the `TableCell` component. For flex-layout cells (e.g. title + badge), add `min-w-0` to the flex container and `truncate` to the text element so the flex child can shrink.

After generating or modifying UI code, run the `design-review` skill to verify the output follows active rules. Fix any violations before considering the work complete.

## Anti-Patterns

Never produce:
- Overused font families (Inter, Roboto, Arial, system-ui)
- Purple gradients on white backgrounds
- Cookie-cutter card grids with identical spacing
- Generic hero sections with stock patterns
- Inline styles for properties the styling framework can handle
- Gray text on colored backgrounds (use a shade of the background color)
- Pure black (#000) or pure white (#fff) as defaults — always tint unless forced-colors/high-contrast mode requires it
- Cards nested inside cards — flatten the hierarchy
- Bounce/elastic easing as a default — feels dated unless used for gesture-driven interactions (drag-to-dismiss, flick, playful moments)
- `position: absolute` dropdowns inside `overflow: hidden` — use `position: fixed`, portals, or `popover`
- `outline: none` without a `:focus-visible` replacement
- Placeholder text as the only label — placeholders disappear on input
- Monospace typography as lazy shorthand for "technical" vibes
- Large rounded-corner icons above every heading — templated, adds no value

→ *See also [nextjs-tailwind reference](references/nextjs-tailwind.md) for Next.js/Tailwind-specific anti-patterns.*
